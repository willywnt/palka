# Cloudflare R2

Olshop stores recording files in Cloudflare R2. Browser uploads go **directly to R2** via presigned PUT URLs.

## Bucket structure

Recordings bucket — keys are `{user_id}/{year}/{month}/{file}` (no bucket/env prefix):

```
cm123abc/2026/05/rec_20260527_a1b2c3d4.webm
```

Products bucket — keys are `{user_id}/{file}`:

```
cm123abc/img_a1b2c3d4.webp
```

Recording keys are generated by `generateStorageKey()`, product-image keys by
`generateProductImageKey()` in the storage module.

## Setup

1. Cloudflare Dashboard → R2 → Create bucket
2. Create API token with **Object Read & Write** on the bucket
3. Set environment variables:

```
R2_ACCOUNT_ID=your-account-id
R2_ACCESS_KEY_ID=...
R2_SECRET_ACCESS_KEY=...
R2_RECORDINGS_BUCKET_NAME=olshop-recordings
# Recordings bucket's own public base (objects served at root, no bucket prefix)
R2_PUBLIC_URL=https://pub-xxxxxxxx.r2.dev
R2_PRODUCTS_BUCKET_NAME=olshop-products
# Products bucket's own public base (objects served at root, no bucket prefix)
R2_PRODUCTS_PUBLIC_URL=https://pub-yyyyyyyy.r2.dev
```

### Public URL options

Each bucket has its **own** public base — `<base>/<key>`, with **no bucket name in the path**.
An `*.r2.dev` subdomain or a custom domain is bound to a single bucket and serves its objects
at the root, so do **not** prepend the bucket name.

| Option               | Use case                                   |
| -------------------- | ------------------------------------------ |
| `*.r2.dev` subdomain | Quick public read access (per bucket)      |
| Custom domain        | Production CDN URL (per bucket)            |
| S3 API endpoint      | Upload only — **not** for browser playback |

Product images need the products bucket's **public access enabled** (gives the `*.r2.dev`
subdomain to put in `R2_PRODUCTS_PUBLIC_URL`). The recordings bucket should stay **private**
(playback uses presigned GET URLs), so its `R2_PUBLIC_URL` is only the stored reference base.

For playback, use presigned GET URLs (already implemented) or a public r2.dev/custom domain.

## CORS configuration

Browser uploads require CORS on the R2 bucket.

### Local development

```bash
node scripts/apply-r2-cors.mjs
```

Uses `scripts/r2-cors.json`:

```json
{
  "AllowedOrigins": ["http://localhost:3000", "http://127.0.0.1:3000"],
  "AllowedMethods": ["GET", "PUT", "HEAD"],
  "AllowedHeaders": ["*"]
}
```

### Production

1. Copy `scripts/r2-cors.production.example.json` and add your domains
2. Apply:

```bash
node scripts/apply-r2-cors.mjs --file scripts/r2-cors.production.json
```

Required origins:

- `https://your-domain.com`
- `https://your-app.vercel.app` (preview)

### CORS rules summary

| Method | Purpose                       |
| ------ | ----------------------------- |
| `PUT`  | Presigned upload from browser |
| `GET`  | Presigned playback / download |
| `HEAD` | Upload verification           |

## Separate buckets per environment

| Environment | Bucket name              |
| ----------- | ------------------------ |
| Local / dev | `olshop-recordings-dev`  |
| Production  | `olshop-recordings-prod` |

Use different API tokens scoped to each bucket.

## Security

- Never expose `R2_SECRET_ACCESS_KEY` to the client
- Presigned URLs expire (upload: 5 min, playback: 1 hour)
- Validate MIME type and file size server-side before presigning
- Storage keys are scoped per environment and user: `{env}/{userId}/{year}/{month}/{filename}`
  where `env` is `production` or `dev` (legacy objects use the `recordings/...` prefix, still
  accepted by the ownership check)

## VPS / MinIO migration (future)

The storage layer uses a `StorageProvider` abstraction (`R2StorageProvider`). To migrate to MinIO or self-hosted S3:

1. Implement `MinioStorageProvider` with the same interface
2. Point `R2_*` env vars at MinIO endpoint (S3-compatible)
3. Keep the same key structure and CORS rules on MinIO

See [vps-migration.md](./vps-migration.md).
